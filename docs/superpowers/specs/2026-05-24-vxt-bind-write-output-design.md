# VXT Bind Write Output Design

Date: 2026-05-24
Status: Draft

## Summary

The initial `bind` package can generate typed Go binding source in memory, but it
still requires callers to write the generated files themselves.

This slice adds a small filesystem-writing layer to `bind` so callers can:

- write generated bindings directly into an explicit output directory
- overwrite regenerated binding files by default
- remove stale files only within the ownership scope of the same binding
- receive a structured write report

This keeps `vxt` library-first and gives `vx` a clean library API to call later.

## Goals

- add an explicit filesystem writer for generated binding output
- make regenerated binding files overwrite by default
- support scoped stale-file cleanup for the same binding
- expose a structured write report
- keep the API library-friendly and explicit

## Non-Goals

- scanning repo roots automatically
- inferring output directories from project layout
- batch generation across many bindings
- cleanup across unrelated generated bindings
- building a generic file-sync engine
- adding CLI behavior to `vxt`

## Public API

Recommended public surface:

```go
type WriteReport struct {
	OutputDir        string
	FilesWritten     []string
	FilesOverwritten []string
	FilesRemoved     []string
}

func WriteOutput(out Output, dir string) (WriteReport, error)

func GenerateToDir(req Request, dir string) (WriteReport, error)
```

Notes:

- `WriteOutput` is canonical
- `GenerateToDir` is a convenience wrapper over `Generate` and `WriteOutput`
- `dir` is always the final output directory, not a repo root

## Output Metadata

To support scoped cleanup safely, generated output should carry ownership
metadata.

Recommended `Output` shape:

```go
type Output struct {
	BindingName string
	PackageName string
	Files       []File
}
```

Recommended `File` shape remains:

```go
type File struct {
	Path    string
	Content string
}
```

`BindingName` is the main binding identity, usually derived from the template
name.

## Ownership Rules

Generated files should be recognized through both:

- naming convention
- explicit header marker

Recommended generated header:

```go
// Code generated by vxt bind. DO NOT EDIT.
// VXT binding: service
// VXT package: servicevxt
```

Recommended file naming convention:

- `<binding>_gen.go`

Both checks must match before a file is treated as owned by the current binding.

## Write Behavior

Default write behavior:

- ensure the output directory exists
- write all generated files into the output directory
- overwrite existing files with the same path
- after writing, reconcile stale owned files for the same binding

`WriteOutput` should:

1. create the output directory if missing
2. write every generated file
3. collect owned stale files in the same directory
4. remove stale files that belong to the same binding but are not part of the
   new output set
5. return a structured `WriteReport`

## Cleanup Scope

Cleanup must be narrow and safe.

Example output directory:

```text
.vxt/
  service_gen.go
  repository_gen.go
  notes.txt
```

If `service` is regenerated:

- `service_gen.go` may be overwritten or removed if stale
- `repository_gen.go` must not be touched
- `notes.txt` must not be touched

The writer must never delete files outside the current binding ownership scope.

## Error Model

Write errors are terminal.

Rules:

- write failure on a target file returns an error immediately
- cleanup failure also returns an error
- partial progress may still appear in the report

This is acceptable because the caller already asked for direct filesystem
effects.

## Why This Boundary

This keeps the contract clean:

- `bind.Generate` remains composable and side-effect free
- `bind.WriteOutput` handles side effects explicitly
- `bind.GenerateToDir` provides convenience without making the library too
  opinionated

This also maps cleanly to future `vx` commands, which can call:

- `bind.GenerateToDir(req, outputDir)`

without reimplementing write or cleanup behavior.

## Success Criteria

- generated bindings can be written directly to an explicit output directory
- regenerated files overwrite by default
- stale files for the same binding are cleaned up safely
- unrelated files in the same directory are preserved
- callers receive a structured write report

## Open Questions

- whether future non-Go generated assets should reuse the same ownership model
- whether write operations should emit optional dry-run support later
- whether the ownership marker should include generator version metadata
