# VXT v0.1.0 Release Readiness Design

## Summary

`vxt` will be prepared for a first public Go package release as `v0.1.0`. This release is intentionally positioned as an experimental public library release, not as a stable `v1` contract and not as a feature-complete implementation of the broader Vandor concept documents.

The design goal is narrow: make `github.com/vandordev/vxt` cleanly consumable through `go get`, understandable on `pkg.go.dev`, and trustworthy for basic library usage around the existing `compile -> validate -> plan -> write` pipeline.

## Release Position

The release will be tagged as `v0.1.0` and described as an experimental public release.

This means:
- the package is meant for real external use
- the package should behave consistently for its documented use cases
- the public API surface should be deliberate and documented
- the maintainers still reserve the right to make controlled breaking changes in later `v0.x` releases

This release does not claim:
- complete alignment with every `vxt` idea in `concept/documents/vxt-specification.md`
- a stable `v1` compatibility promise
- a CLI or executable distribution
- exhaustive performance, benchmark, or fuzz-hardening work

## Scope

### In Scope

- release-readiness of the Go module at `github.com/vandordev/vxt`
- a documented and intentionally bounded public package surface
- documentation sufficient for first-time package consumers
- legal packaging required for public code distribution
- a repeatable verification and tagging workflow for the first release
- an external smoke test that validates the package from another module

### Out of Scope

- implementing missing concept-spec features unrelated to release-readiness
- introducing a `vxt` CLI or binary
- promising `v1.0.0` API stability
- broad refactors that do not improve release quality
- multi-platform CI expansion, benchmark suites, or release automation beyond what is needed for `v0.1.0`

## Public API Shape

The supported public package surface for `v0.1.0` is:
- `github.com/vandordev/vxt`
- `github.com/vandordev/vxt/runtime`
- `github.com/vandordev/vxt/diag`
- `github.com/vandordev/vxt/source`
- `github.com/vandordev/vxt/write`

Everything under `internal/` is implementation detail and may change freely without notice.

### Intended Roles

- `vxt`: conceptual entry point and root package documentation
- `runtime`: primary consumer-facing pipeline for compile, validate, plan, and write operations
- `diag`: structured diagnostic codes and diagnostic values that external tooling can inspect
- `source`: source-loading abstractions used by the runtime pipeline
- `write`: output target abstractions for materializing planned output

### Public API Constraints

- Public packages should expose only types and functions needed for normal library consumption.
- Internal parser, renderer, AST, schema, and planning details should remain hidden unless there is a clear consumer need.
- Exported names in public packages must have package comments or identifier comments suitable for `pkg.go.dev`.
- Public examples and README usage must align with the actual exported API.

## Compatibility Policy

The compatibility policy for `v0.1.0` is intentionally modest:

- Avoid gratuitous breaking changes across `v0.x` releases.
- Allow controlled breaking changes when improving the public API before `v1.0.0`.
- Treat undocumented behavior and all `internal/...` details as unstable.
- Keep diagnostic codes stable once publicly documented unless there is a strong reason to change them.

The release messaging should state clearly that:
- `v0.1.0` is usable
- public packages are intentional
- the API is still evolving

## Release Criteria

The release is considered ready only if all of the following are true:

1. The repository is clean and all tracked changes intended for release are committed.
2. `go test ./...` passes from the module root.
3. `go list ./...` succeeds from the module root.
4. `go doc` for public packages and key exported symbols is readable and free from obvious leakage of internal package types in signatures.
5. README content is sufficient for a first-time consumer to understand installation, supported packages, basic usage, and current limitations.
6. `LICENSE` and `TRADEMARK.md` are present and consistent with the intended distribution model.
7. A smoke test from an external temporary module can fetch and use `github.com/vandordev/vxt` successfully against the intended release revision or tag.
8. A release note exists for `v0.1.0` describing scope, supported packages, notable behavior, and non-goals.

## Deliverables

The first public release should ship with these deliverables:

- Git tag: `v0.1.0`
- release notes for `v0.1.0`
- finalized README for public package consumers
- `LICENSE`
- `TRADEMARK.md`
- an explicit release verification procedure that can be repeated before tagging
- a small external smoke test fixture or documented smoke test command sequence

Optional but useful:
- a short changelog section or release summary block in the README

## Verification Workflow

The release workflow should remain simple and reproducible:

1. Verify the working tree is clean.
2. Run repository verification commands from the module root:
   - `go test ./...`
   - `go list ./...`
3. Inspect package documentation for public surface clarity:
   - `go doc github.com/vandordev/vxt`
   - `go doc github.com/vandordev/vxt/runtime`
   - targeted `go doc` calls for key exported types and functions
4. Run an external smoke test from a separate temporary module that imports `github.com/vandordev/vxt` and exercises the basic runtime pipeline.
5. Prepare or review release notes for accuracy.
6. Create tag `v0.1.0`.
7. Perform a final post-tag sanity check that the tagged revision still passes the same verification steps.

The workflow should be documented so that future `v0.x` releases can reuse it with minimal changes.

## Smoke Test Expectations

The external smoke test should validate the real consumer path rather than only in-repo behavior.

Minimum expectations:
- initialize a fresh temporary Go module outside the `vxt` repo
- add a dependency on `github.com/vandordev/vxt`
- compile and run a small program using documented public APIs
- confirm basic compile or runtime success for a known-good input path

The smoke test does not need to be large. Its purpose is to catch:
- broken module path assumptions
- accidental dependence on internal packages
- missing docs-driven imports
- public API signatures that are awkward or impossible to consume externally

## Documentation Requirements

The release documentation should make these points obvious:

- what `vxt` is
- what `v0.1.0` supports today
- which packages are intended for public use
- a minimal example of using the runtime pipeline
- that the project is still pre-`v1`
- how licensing and branding are separated

The README should optimize for first-use clarity, not for documenting every internal subsystem.

## Risks

### Risk: Public API looks stable but is not actually deliberate

Mitigation:
- keep the public package list short
- hide unstable implementation details behind `internal/`
- state the `v0.x` compatibility policy explicitly

### Risk: `go get` works locally but not as a real external consumer experience

Mitigation:
- require an external smoke test before tagging
- avoid treating in-repo tests as sufficient evidence of release readiness

### Risk: Release messaging over-promises completeness

Mitigation:
- position the release as experimental
- document non-goals and current limitations clearly

### Risk: Documentation drifts from actual API

Mitigation:
- validate examples against the real exported API
- treat README and `go doc` clarity as part of the release gate

## Non-Goals

This release design explicitly does not attempt to:
- settle the final `v1` package structure
- expose every internal concept as public API
- deliver the entire Vandor product vision
- solve ecosystem packaging for `vx` or `vpkg`
- provide production-grade governance, CI, or automation beyond what `v0.1.0` needs

## Implementation Units

The release-readiness work should be planned around a few clear units:

- documentation unit: README, release notes, and public package docs
- verification unit: repeatable local verification and external smoke test
- release mechanics unit: tagging procedure and release checklist

Each unit should remain independently understandable and testable so the implementation plan can decompose work cleanly.

## Success Criteria

This design is successful when:
- an external Go user can discover the module, read the docs, and import the intended packages without confusion
- the repo communicates clearly that `v0.1.0` is experimental but real
- the first release can be repeated from a documented checklist instead of tribal knowledge
